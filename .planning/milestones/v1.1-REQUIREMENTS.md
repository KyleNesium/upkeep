# v1.1 Requirements — Update Skill Overhaul (Mac-only)

> Goal: make `/upkeep:update` the best macOS update orchestrator on GitHub —
> fast, transparent, compatibility-aware, and walk-away-friendly.

## Stakeholder context

Driven by a live run on 2026-05-07 against Kyle's laptop. The current skill
worked but exposed a set of papercuts (sequential discovery, no version
deltas, no cross-manager compatibility, per-category gate fatigue) documented
in detail in `.planning/research/v1.1-pain-points.md`.

## Functional requirements

| # | Requirement | Acceptance |
|---|---|---|
| R1 | Discovery runs in parallel via specialized agents | Wall time ≤ 50% of v1.0 sequential discovery on a populated machine |
| R2 | Each scout emits structured JSON, not free-form prose | Synthesizer can consume without re-parsing |
| R3 | Compatibility synthesizer flags cross-manager risk | brew:node ⇒ npm-globals rebuild; brew:python ⇒ pipx; brew:openssl ⇒ ruby native ext; brew:ruby ⇒ gems |
| R4 | Major-version bumps surfaced separately from patch bumps | Top-of-report "⚠ major bumps" section before the per-tool table |
| R5 | Single approval gate (or per-category opt-out) | One AskUserQuestion covers all categories; user can drop categories without re-prompting per-tool |
| R6 | Apply phase parallelizes independent ecosystems | npm + pipx + gems run concurrently; brew alone; mas/macOS last |
| R7 | PATH shadowing detection | After every brew upgrade, `which -a <bin>` is checked for duplicates and surfaced |
| R8 | macOS system Ruby detection | `gem update` auto-uses `--user-install` when system Ruby (2.6) is in use |
| R9 | Post-flight health check | brew doctor (silent unless warnings), command-v re-resolution, deprecation summary |
| R10 | Time estimate in plan | Heuristic ETA shown before approval gate |
| R11 | Disk-space pre-flight | Refuse to start if free space below 5 GB; warn below 10 GB |
| R12 | Codex skills git-pull where applicable | `~/.codex/skills/*/.git` repos are treated like Claude skills |
| R13 | Audit / skills / packages / all sub-modes preserved | Backwards-compat with v1.0 invocations |
| R14 | Linux & WSL2 paths untouched | All v1.0 OS_TYPE branching for linux/wsl2 still works |

## Non-functional requirements

| # | Requirement |
|---|---|
| N1 | No additional external dependencies (no npm install, no python venv) — pure shell + Claude tool calls |
| N2 | Skill remains a single SKILL.md file (with optional sibling JSON data file for compatibility matrix) |
| N3 | All agent prompts are self-contained — no implicit context sharing |
| N4 | Failures in one ecosystem never block another |
| N5 | Every destructive command remains gated; rules in v1.0 SKILL.md preserved |

## Out of scope (deferred to v1.1.x or later)

- Linux/WSL2 port of the new architecture
- Plugin-cache (Claude Code) self-update — Claude Code manages this
- Codex marketplace plugin self-update — manual remains
- Automated scheduling (cron) — out of scope
- GUI / TUI — terminal only
- Rollback automation (snapshot before upgrade) — manual rollback hints only

## Success metrics

- Live run on a populated machine with 30+ brew outdated, ≥1 skill behind,
  3+ language ecosystems behind: complete in under 50% of v1.0 wall time
  (excluding actual download/compile time).
- Zero "had to ask the user something they could have answered up-front."
- All compatibility-flag scenarios from R3 fire on a synthetic test machine.
