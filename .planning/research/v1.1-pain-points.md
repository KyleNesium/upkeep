# v1.1 Research — Pain Points from 2026-05-07 Live Run

Captured during a real `/upkeep:update` run on Kyle's M-series Mac. Used as the
input to v1.1 requirements.

## Summary

Skill worked end-to-end and produced an accurate report. Friction points were
all in *speed*, *transparency*, and *cross-manager awareness*.

## Detailed findings

### 1. Sequential discovery is slow

11 separate bash calls for discovery (`brew outdated`, `npm outdated -g`,
`pipx list`, `gem outdated`, `rustup check`, `cargo install-update --list`,
`uv self version`, `bun --version`, `deno --version`, `mise outdated`, `mas
outdated`, `softwareupdate -l`). Total ~30s wall time even when most are
no-ops because every call goes round-trip through the assistant.

**Fix:** four parallel agents each owning one domain. Each agent runs its
section's bash calls and returns structured JSON.

### 2. No version deltas in overview

Showed "36 outdated" without saying which package, what version delta. User
can't gauge whether to approve. Especially relevant for major bumps
(node 20 → 22, python 3.12 → 3.13).

**Fix:** discovery agents capture version pairs; synthesizer sorts by bump
severity (major / minor / patch) and shows top-10 by severity.

### 3. No cross-manager compatibility awareness

`brew upgrade` ran first, upgrading `node`. Then `npm update -g` ran with 6
deprecation warnings about packages that need rebuild against the new node.
No flag to user that "after this brew upgrade, these npm globals may need
reinstall."

**Fix:** compatibility matrix:
- brew:node → flag npm globals
- brew:python → flag pipx, uv tools
- brew:openssl → flag ruby gems w/ native ext (nokogiri, openssl gem)
- brew:ruby → flag user gems
- brew:cmake / ffmpeg → no downstream

### 4. PATH shadowing silently surfaced

brew warned: "gemini-cli executables are shadowed by other commands earlier in
your PATH: gemini (shadowed by /Users/kyle/.superset/bin/gemini)". Skill
captured the line in stdout but didn't surface or interpret it.

**Fix:** post-apply `which -a` check for every upgraded brew formula's
binaries.

### 5. macOS system Ruby gem update fails without `--user-install`

Default `gem update` writes to `/Library/Ruby/Gems/2.6` which requires sudo
on modern macOS. The user spec says "never run sudo" — so `gem update` would
fail. I had to manually swap to `gem update --user-install`.

**Fix:** synthesizer detects `which ruby` resolves to system Ruby 2.x → add
`--user-install` flag automatically.

### 6. 31-commit gstack changelog dumped raw

User saw 31 commit subjects in the gate prompt. Hard to parse for breaking
changes.

**Fix:** synthesizer scans CHANGELOG.md for `### Breaking` / `BREAKING`
sections between current and target versions; surfaces those as a separate
list above the commit dump.

### 7. Per-category gate fatigue

Brew → npm → pipx → gems → uv → bun = 6 separate Y/N prompts. User can't walk
away. Long brew compile in middle means revisiting the session.

**Fix:** single gate with full plan; user can opt-out specific categories
inline.

### 8. No time estimate

"Upgrade Homebrew (36 outdated packages)" with no idea if that's 2 minutes or
40 minutes. brew upgrade in this run took ~10 minutes; gem update took ~12
minutes. ETA helps user decide whether to start now or defer.

**Fix:** historical heuristic — store last run's wall time per category in
`~/.claude/data/upkeep-history.json`; show median.

### 9. No disk-space pre-flight

brew upgrade with mlx + ffmpeg-full + qt + python wheels can need 5-10 GB
temporarily. Failure mid-run is messy.

**Fix:** check `df -k /` for free space before starting; warn below 10 GB,
refuse below 5 GB.

### 10. Codex skills marked "manual update required"

`~/.codex/skills/*` are often git repos too. Skill listed 16 codex skills as
manual, never tried git-pull.

**Fix:** skills-scout treats `~/.codex/skills/*/.git` exactly like Claude
skills.

### 11. "Managed by plugin manager" is a dead end

upkeep itself is installed via Claude Code's plugin system. Skill correctly
detected this and skipped, but offered no path forward — user has to know to
run `/plugin update`.

**Fix:** when plugin-managed, surface the exact command (`/plugin update
upkeep`) as a Manual Steps line.

## Architecture decision: parallel agents vs single big bash

Considered: one big bash script with `&` for parallelism. Rejected:
- Bash background jobs are hard to track from inside a Claude skill (no clean
  way to wait + collect structured output)
- Each parallel section needs domain-specific reasoning (e.g., "is this brew
  formula a build dep or a runtime dep?") that benefits from per-agent
  context isolation
- Compatibility synthesis is naturally an LLM task — give it the JSON, get
  the plan

Going with four parallel `Agent` calls in a single tool-use block, each with
a tight prompt and a structured-output instruction.

## Architecture decision: Mac-only first

User explicitly requested. Rationale:
- Linux/WSL2 logic in current `update` skill is fresh (v1.0 just shipped)
- Mac is where the test machine lives — fastest iteration loop
- Compat matrix differs by OS (brew:openssl on Linux is unusual)
- Once macOS proves the approach, port to Linux in v1.1.1
